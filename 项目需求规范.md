# R_U_OK 项目需求文档

版本：v0.1  
日期：2026-07-06  
项目类型：Hackathon MVP / 医疗器械本地合规智能自查 Agent

产品名：R_U_OK  
口号：问己合规，预见风险

## 1. 项目概述

R_U_OK 是一个面向医疗器械研发、供应链、生产、质量、注册等职能团队的本地合规智能自查 Agent。它通过只读基础知识库、LLM wiki 和用户本地项目资料的关联分析，帮助业务团队在正式审核、质量评审、注册提交或项目交付前，回答一个核心问题：

> 我们现在 OK 吗？要求是否覆盖、证据是否充分、风险是否可见？

R_U_OK 不替代审核员、QA 批准、RA 判断或任何正式合规结论。它的定位是通过知识库赋能和场景化工作流，把分散的法规标准、内部要求、项目文档、报告和历史质量信号整理成可追溯、可复核、可行动的风险提示、预警和自查建议。

## 2. 名称解释

R_U_OK 表面含义是 `Are you OK?`，用于表达项目在正式评审前的 readiness check。

项目语境下建议解释为：

- **R = Regulatory / RA / Requirement / Risk**：法规注册要求、RA 关注点、基础条款要求和风险信号。
- **U = User / Uploaded Evidence / Unit**：使用者所属业务职能、用户上传的本地项目资料和具体业务单元。
- **OK = Objective Knowledge Check**：基于知识库、证据链和可引用来源的客观检查。

一句话解释：

> R_U_OK = Regulatory/RA requirements + User evidence + Objective Knowledge check.

## 3. 背景与业务痛点

国内医疗器械行业受 NMPA、ISO 13485、相关法规及产品标准严格约束，研发、供应链、生产、质量、注册等全流程合规要求繁杂。日常合规落地普遍存在前置风控不足、专业理解差异、信息分散割裂、跨团队协同低效四大痛点。

1. **合规自查属于滞后措施，前置风控缺失**  
   许多问题是在正式审核、质量评审、注册提交或项目交付前才被集中发现，风险提示没有前移到日常项目执行过程。

2. **自查过度依赖 Q 部门或个人经验**  
   不同职能对法规、SOP、产品标准和证据要求的理解不一致，业务团队难以形成统一的合规判断框架。

3. **信息分散割裂，检索成本极高**  
   法规标准、内部要求、验证资料、技术报告、CAPA、偏差、投诉、供应商记录、培训记录和历史审核发现分散在不同文件和系统中，人工检索、比对和关联耗时高。

4. **跨团队合规沟通成本高、协同低效**  
   研发、供应链、生产、质量、注册等团队关注点不同，缺少统一的风险语言、证据链视图和任务看板，导致问题沟通和整改准备效率低。

这些问题会导致准备效率低、自查质量不稳定、跨部门理解不一致，并增加正式审核、注册提交或质量评审中的返工风险。R_U_OK 要解决的不是“替业务做最终判断”，而是让风险提示更前置、证据链更清楚、职能间沟通更高效。

## 4. 产品目标

### 4.1 业务目标

- 帮助业务团队在正式评审前快速识别合规缺口、证据不足和风险关注点。
- 将基础法规/标准要求与项目资料证据建立可追溯关联。
- 将历史发现项和内部质量信号转化为自查计划候选和风险看板。
- 按任务场景生成自查重点、报告检查项、证据链路和风险提示，提升全员质量意识。
- 以本地优先方式处理项目资料，降低敏感资料外流风险。

### 4.2 MVP 目标

- 支持只读基础知识库浏览和图谱展示。
- 支持项目资料或样例数据形成 LLM wiki / evidence view。
- 支持根据基础知识库生成风险关注点和自查计划候选。
- 支持通过 Kanban 展示发现项、自查计划、风险焦点和证据准备状态。
- 支持 Chat/RAG 形式回答自查问题，并给出引用来源和人工复核提示。

### 4.3 初步解决方案

R_U_OK 采用“知识库赋能 + 场景化工作流”的方案，不替代人工终审与合规判定，只做风险提示和预警。

1. **采用 LLM wiki 的知识库管理思路**  
   将法规标准、内部要求、项目文档、报告和历史质量信号结构化为可追溯的知识节点与关系链路，弥补单纯 embedding 检索容易遗漏上下文关联的问题。

2. **整合基础要求与项目资料**  
   基础层包含法规、国家标准、ISO 13485、SOP、模板和内部要求；项目层包含验证资料、技术报告、设计变更、CAPA、偏差、投诉、供应商记录和注册资料等。

3. **按任务场景生成自查重点**  
   系统根据用户当前任务背景和职能场景，生成自查重点、报告检查项、证据清单、风险提示和人工复核建议。

4. **展示证据链路与风险看板**  
   通过 LLM wiki、Graph 和 Kanban 展示“要求 -> 证据 -> 风险 -> 控制 -> 自查计划”的链路，帮助团队高效锁定任务背景并提升跨团队协同。

## 5. 目标用户与场景

### 5.1 研发 / R&D

典型任务：验证报告、技术报告、设计变更、风险分析提交前自查。

核心问题：

- 需求、测试、风险、偏差和结论之间是否可追溯？
- 报告结论是否有足够证据支持？
- 是否存在引用缺失、证据不足或结论过强？

预期输出：研发报告 reviewer checklist、证据缺口、风险控制闭环提示。

### 5.2 生产 / Manufacturing

典型任务：生产现场、批记录、培训、设备、物料和关键工艺参数自查。

核心问题：

- 高风险基础要求是否被日常执行覆盖？
- 批记录、设备状态、人员培训和物料状态是否有证据？
- 历史偏差或审核发现是否可能重复出现？

预期输出：现场自查清单、批记录完整性检查、关键风险提醒。

### 5.3 质量 / QA

典型任务：CAPA、偏差、投诉、变更、审核发现趋势分析和自查计划制定。

核心问题：

- 哪些问题具有 recurring 或 systemic 信号？
- 哪些风险应进入近期自查或管理层关注？
- 哪些证据和控制措施需要提前准备？

预期输出：质量风险摘要、自查计划候选、管理层 summary、风险看板。

### 5.4 注册 / RA

典型任务：注册资料、技术文件、声明和法规条款覆盖检查。

核心问题：

- 注册资料是否覆盖关键法规和标准要求？
- claims 与证据是否一致？
- 哪些高风险表述需要人工确认？

预期输出：注册资料 completeness checklist、claims consistency review、引用证据清单。

### 5.5 供应链 / Supplier Quality

典型任务：供应商质量问题、来料异常、供应商变更、质量协议和供应商审核准备。

核心问题：

- 哪些供应商、物料或变更记录存在高风险信号？
- 来料检验、供应商资质、CoA、质量协议等证据是否完整？
- 供应商问题是否与内部偏差、投诉或 CAPA 形成重复风险？

预期输出：高风险供应商/物料提示、供应商自查清单、来料证据要求、供应商变更影响分析 draft。

## 6. 产品范围

### 6.1 MVP 范围内

- 基础知识库：NMPA、ISO 13485、法规、国家标准、产品标准、SOP/政策、条款、义务、风险、控制措施。
- LLM wiki：将基础要求、项目资料、历史质量信号和证据记录组织为可追溯知识节点与关联链路。
- 业务信号：内部/外部审核发现、自查发现、客户审核、偏差趋势、投诉趋势、培训缺口等 findings。
- 自查计划：根据 findings、risks、controls、clauses 生成 audit_items 候选。
- 看板：展示发现项输入池、自查计划候选、公开风险关注点、证据和控制准备。
- Chat 问答：基于知识库回答自查问题，保留引用和人工复核提示。
- 本地运行：应用、基础库和项目工作区均在本机运行和保存。

### 6.2 MVP 范围外

- 不做正式审核计划批准。
- 不输出最终合规结论。
- 不替代 QA/RA/审核员签字或审批。
- 不直接连接生产系统或质量系统进行自动写回。
- 不处理未脱敏的真实敏感数据作为公开演示材料。

## 7. 核心用户流程

### 7.1 基础知识库查看

1. 用户打开 Dashboard 查看基础知识库规模和实体类型。
2. 用户打开 Graph 查看条款、义务、风险和控制之间的关系。
3. 用户查看某一条款或风险的来源、引用和关联控制。

### 7.2 项目资料自查

1. 用户选择职能和任务场景，例如研发验证报告自查或生产现场自查。
2. 用户上传或指定本地项目资料文件夹。
3. 系统解析资料并形成项目 wiki / evidence records。
4. 系统将项目证据与基础知识库要求进行关联。
5. 系统输出缺口、风险等级、证据清单和人工复核建议。

### 7.3 Findings 驱动自查计划

1. 用户导入或录入内部/外部审核发现、偏差趋势、投诉趋势等 findings。
2. 系统识别相关条款、义务、风险和控制措施。
3. 系统生成 audit_items 候选，说明为什么要查、查什么、需要什么证据。
4. 用户在 Kanban 上查看优先级、责任人、状态和证据准备情况。
5. 负责人进行人工确认、调整和正式发布。

### 7.4 Chat 自查问答

1. 用户输入问题，例如“近期自查应该优先关注哪些主题？”
2. 系统检索基础库、项目证据、findings 和 audit_items。
3. 系统给出结构化回答，包括建议主题、依据、风险、所需证据和人工复核提示。
4. 每条建议应尽量保留来源引用。

## 11. 合规与责任边界

R_U_OK 的输出应使用以下口径：

- “初步自查建议”
- “风险关注点”
- “证据准备提示”
- “人工复核建议”
- “自查计划候选”
- “风险提示和预警”

R_U_OK 不应使用以下口径：

- “本项目已合规”
- “无需 QA/RA 审核”
- “验证已正式通过”
- “资料可直接提交”
- “最终审核结论”

所有正式判断、批准、对外提交和审核结论仍由具备职责和资质的负责人完成。

## 12. MVP 演示路径

推荐演示顺序：

1. 打开 Dashboard，展示基础知识库规模。
2. 打开 Graph，展示 clause、obligation、risk、control 的关系。
3. 打开 Kanban，展示风险点、自查计划候选和证据准备状态。
4. 在 Chat 中提问：“根据当前风险图谱，近期自查应该优先关注哪些主题？”
5. 展示一个 finding 如何驱动 audit_item，并说明它只是计划候选。
6. 强调本地运行、基础库只读、项目资料隔离和人工复核边界。

## 15. 风险与待确认问题

### 15.1 风险

- 文档解析质量不足，导致证据关联不稳定。
- LLM 输出过度确定，容易越过合规边界。
- 基础知识库样例不足，演示效果不够真实。
- 用户误解为正式审核或自动批准工具。
- 本地部署包、模型和依赖环境可能影响演示稳定性。

### 15.2 待确认问题

- Hackathon 演示是否使用完全脱敏的真实样例，还是人工构造样例？
- MVP 优先职能是否锁定 R&D / QA / RA 三类？
- 本地 LLM 模型、企业内模型或在线模型的可用边界是什么？
- 基础知识库由谁维护、如何发布版本？
- Kanban 是否需要导出为 Excel / PDF / PPT？
